## Bug Fix: AlignArrangePlugin 排列漂移 (两连修)

**修复1 - computedPosition 坐标系 (不完整):**
`arrange()` 用 `computedPosition`（绝对坐标）读取但用 `updateNode({ position })`（相对坐标）写回。当节点在群组内时坐标系不一致导致漂移。
→ `AlignArrangePlugin.ts:41-42`: 移除 computedPosition 回退，始终用 `n.position`

**修复2 - VueFlow nudge 冲突 (真正根因):**
VueFlow 内置方向键微移 (`arrowKeyDiffs`)，在 `NodesSelection.onKeyDown` 和 `NodeWrapper` keydown 中不检查 Ctrl 修饰键。事件冒泡时 VueFlow 子元素的 handler 先于 ShortcutManager（document 级）触发，导致每次 Ctrl+Arrow 排列前 VueFlow 先把选中节点微移 1~2px（两个 handler 各移一次）。
→ `AlignArrangePlugin.ts:61-80`: 添加 document 级 capture 阶段 keydown 监听器，检测 Ctrl+Arrow 时 `stopImmediatePropagation()` 阻止事件传播到 VueFlow 子元素。Capture 阶段拦截意味着事件不下降、不冒泡，VueFlow 和 ShortcutManager 都不会再看到这个事件。

- **问题**: `context.panels.useValue("align-arrange.gap", context.store, config.gap)` 报 `TypeError: Cannot read properties of undefined (reading 'align-arrange')`
- **根因**: `context.store` 是 `CanvasStoreAPI`（有 get/set/watch/getState 方法），而 `PanelRegistry.useValue()` 期望 `store` 是 `{ core, plugins }` 结构。`context.store` 是 truthy，所以 `PluginContext` 中 `store ?? canvasState` 的 fallback 不触发，CanvasStoreAPI 被直接传入 PanelRegistry，其 `s.plugins` 为 undefined。
- **修复**:
  1. `AlignArrangePlugin.ts:24` — 将 `context.store` 改为 `null as any`，让 PluginContext 自动降级到 canvasState
  2. `PluginContext.ts:401` — 增强 useValue 包装：检测 store 是否有 `plugins` 属性，没有则降级到 canvasState，防止其他插件再踩坑

## Refactor: context.store.toRef — 统一插件 store

- **背景**: 用户指出 `context.store` (CanvasStoreAPI) 和 `useValue` 需要的 store ({ core, plugins }) 是两套重叠概念，容易用错
- **方案**: 在 `CanvasStoreAPI` 上加 `toRef<T>(key, defaultValue): Ref<T>`, 内部用 `computed` 读写 `canvasStore.state.plugins[pluginName][key]`
- **改动文件**:
  - `types.ts`: CanvasStoreAPI 增加 `toRef` 签名
  - `PluginContext.ts`: createPluginStore 实现 `toRef`, 补充 `computed` / `Ref` import
  - `AlignArrangePlugin.ts`: `context.panels.useValue(...)` → `context.store.toRef('gap', config.gap)`, 移除无用 `watch` import
  - `MiniMapPlugin.ts`: 5 处 `context.panels.useValue(...)` → `context.store.toRef(...)`
- **效果**: 插件不再需要传外层 store, `toRef` 自动走正确的命名空间; `context.store` 成为唯一的状态入口
- **清理**: 移除 `PluginContext.panels.useValue` 包装（无人调用），`PluginContext.panels` 类型改为 `Omit<PanelRegistryAPI, 'useValue'>`；
  `PanelRegistry` 类自身保留 `useValue`（Canvas.vue 的 panel UI 层直接调用）
- **进一步清理（useValue 彻底下线）**:
  - `PanelRegistryAPI` 接口删除 `useValue` 签名 → `types.ts` 中 `Omit<>` 回退为纯 `PanelRegistryAPI`
  - `PluginManager.ts` stub context 删除 `useValue: () => null`、补充 `toRef` stub
  - `PluginContext.ts` 删除 `canvasState` 参数、去除重复 `Ref` import
  - `Canvas.vue` 删除 `canvasState` 传参

## Bug Fix: 吸附功能失效 — eventBus 未传入 composable

- **问题**: 重构后单节点拖拽连接线到节点上，3D 动效有但吸附不生效（不创建连线）
- **根因**: `useCanvasConnection` 创建时未传 `eventBus`，导致 `eventBus?.emit('connect', connection)` 是 no-op。`ContextMenuPlugin` 监听 `connect` 事件来设置 `lastNativeConnectAt` 时间戳，收不到事件则时间戳保持 0。随后插件的 `connectEnd` handler 检查 `Date.now() - 0 < 80` 不通过，不提前返回，覆盖了 composable 设置的 `hoverTarget`/`snapTarget`，甚至弹出菜单/创建临时连接
- **修复**: Canvas.vue 中 `const manager = new PluginManager()` 提前到 composable 之前，传入 `eventBus: manager.eventBus`
- **事件流**: composable `onConnectEnd` 创建连接 → `eventBus.emit('connect')` → 插件 `connect` handler 设 `lastNativeConnectAt = Date.now()` → 插件 `connectEnd` handler 检查 `< 80ms` → true → 提前返回，不干扰

## Bug Fix: 批量连线吸附 — 视觉 snap + createConnection 调查

- **问题**: 多选拖拽到吸附区域有 3D 动效（hoverNode valid），但松开后"没有吸附上去"
- **日志证据**: `onBatchConnectEnd` 中 `findNearestValidTarget` 成功返回目标节点；第一条线 `createConnection` → `result: true`；第二条线 `result: false`
- **视觉问题**: `updateBatchTempTarget` 只跟鼠标走，缺少 snap 到端口位置的逻辑（单条连线在 `buildConnectionEdgeProps` 中有 snap）
- **修复1**: `updateBatchTempTarget` 加入 snap 逻辑 — `findNearestValidTarget`/`findNearestValidSource` 找到目标时，临时节点位置 snap 到目标端口位置
- **调查中**: 在 `createConnection` 和 `isValidConnection` 中加详细日志，定位第二条 `result: false` 的精确原因（可能是去重/循环检测/端口缺失）

## Bug Fix: 节点主体不应 snap

- **问题**: `updateBatchTempTarget` 中加了 `findNodeBodyAtPoint` 作为 snap 回退，导致鼠标在节点主体上时临时线也 snap 到端口
- **根因**: 节点主体和吸附区域是两个独立区域——吸附区域 snap 位置，节点主体只产生 3D 反馈不 snap
- **修复**: `updateBatchTempTarget` 删除 `findNodeBodyAtPoint` 回退，只用 `findNearestValidTarget`/`findNearestValidSource`。`findNodeBodyAtPoint` 只在 `updateBatchConnectFeedback`（3D 反馈）和 `onBatchConnectEnd`（松开判定）中使用
