## 修复 ImageCropper 裁剪框 resize 弹回 + 不显示问题

**根因（systematic-debugging 定位）：**
1. ImageCropper.vue 从未被渲染（ImageNode.vue 没 import 它）
2. resize 公式完全错误：`nw = Math.abs((scX + dx) - ax)` 漏了 `scW`，导致拖拽时宽度变成 `|dx|`（几像素），被 clamp 强制拉回 MIN_CROP=20 → "弹回"
3. `containerW/containerH` 用 side-effect ref 写入（在 wrapperStyle computed 内赋值），zoom 变化时 display.scale 过期

**修复：**
- `ImageNode.vue`：`import ImageCropper`，`v-if="isCropping"` 渲染，传入 nodeId/imageUrl/imageWidth/imageHeight
- `ImageCropper.vue` `onPointerMove` resize 逻辑重写：固定对边、移动当前边，`nw = scW + dx`（e/s）或 `nw = right - nx`（w/n）
- `containerW/containerH` 改为独立 computed
- `z-index` 从 +10 改为 +100
- 去掉 `overflow: hidden`（裁切手柄不被裁切）
- 测试文件：`crop-test.html`（含自动化断言）

## 分析：节点 Toolbar 插槽的自定义机制

- **CustomNode.vue（19-34）** 是 vue-flow `type=custom` 的统一外壳，3 个插槽 `top-toolbar`/`content`/`bottom-toolbar` 转发给 BaseNode；toolbar 插槽默认渲染 `BaseToolbar`
- **节点内容组件**（StageNode.vue/ImageNode.vue 等）只负责 `content` 区，不直接控制 toolbar
- **数据驱动是标准方式**：在 `nodes/<type>/<Type>NodePlugin.ts` 的 `install()` 中 `context.toolbars.register()` 注册 `ToolbarButtonDefinition`，配套 `context.commands.register()` 注册命令；BaseToolbar 按 `position`+`nodeTypes` 过滤渲染
- `ToolbarButtonDefinition` 支持字段：icon(SVG串/组件)、title、tooltip、position、nodeTypes、order、dropdown、customRender(完全替换按钮UI)、visible/disabled(可函数,接收CommandContext)
- 现有节点全部走方式A（register 按钮），image 的 upload 按钮用了 customRender=ImageUploadButton
- 插槽覆盖（方式C）当前无人使用——CustomNode 由 vue-flow 自动渲染，无上层模板填充 slot

**设计思路：** 用一个 `data._overlay` 对象统一管理所有临时覆写状态（`_cropMode`、`_toolbarGroup` 等），退出时 `delete data._overlay` 一步恢复所有状态，不需要逐字段清理。

- `src/canvas/core/nodes/image/ImageNodePlugin.ts` — 核心改造

**ImageNodePlugin 改动：**
- `handleImageCrop`：进入裁剪设 `_overlay = { _cropMode: true, _toolbarGroup: 'crop' }` + 初始化 `_cropRect`
- `handleImageCropConfirm`：读 `_cropRect` → 按裁后比例更新 cardWidth/cardHeight → `delete _overlay + _cropRect + _cropMode`
- `handleImageCropCancel`：`delete _overlay + _cropRect + _cropMode`（不改变数据）
- 新增命令：`image.cropConfirm`、`image.cropCancel`
- 新增 toolbar 按钮：确认/取消，标 `group: 'crop'`，`visible` 检查 `_overlay._cropMode`

**状态流：**
- 正常 → 点裁剪 → `_overlay = { _cropMode: true, _toolbarGroup: 'crop' }` → toolbar 切换为确认/取消 + 裁剪 UI 显示
- 裁剪中 → 点取消 → `delete _overlay` → toolbar 恢复 default 组 + 裁剪 UI 消失
- 裁剪中 → 点确认 → 获取 _cropRect → 更新 cardWidth/Height → `delete _overlay`
- vue-tsc 通过

## 裁剪确认后创建新图片节点

**`handleImageCropConfirm` 改造为 async**，完整流程：
1. 退出裁剪模式（delete `_overlay`）
2. 用 Canvas API 裁剪原图：`new Image()` 加载 → `canvas.drawImage(sx,sy,sw,sh,0,0,dw,dh)` → `toBlob('image/png')`
3. 尝试持久化到 assetManager
4. 计算卡片尺寸（按 MAX_PREVIEW 比例）
5. `vf.addNodes` 在源节点右侧 40px 处创建新图片节点，携带裁剪后的 blob URL + 尺寸数据

## 修复裁剪新节点图片不显示

**根因：** `canvas.toBlob() + URL.createObjectURL(blob)` 在某些环境下异步时序不可靠。
**修复：** `imageUrl` 改用 `canvas.toDataURL('image/png')`（同步 data URL，`<img src>` 立即可用）；`toBlob` 仅用于 assetManager 持久化。补充 `label`、`sourcePosition`/`targetPosition`，对齐 FileDropPlugin 节点结构。

## Canvas 裁剪图片的终极修复：createImageBitmap 两步解码

**问题：** 连续 5 轮尝试，canvas drawImage(HTMLImageElement) / drawImage(ImageBitmap) / decode() / onload+DOM挂载 全部画不出像素（testPixel 全 0）。

**最终方案：**
```
fetch(imageUrl) → rawBlob                                 // 拉取二进制
createImageBitmap(rawBlob) → fullBitmap (576×944)         // 完整解码
createImageBitmap(fullBitmap, sx, sy, sw, sh) → cropBitmap // 从 bitmap 裁剪
canvas.drawImage(cropBitmap, 0, 0)                         // 画到 canvas ✓
```

**关键发现：**
- `createImageBitmap(blob, sx, sy, sw, sh)` 直接从 blob 裁剪 → 0×0（浏览器 bug）
- 先完整解码再裁剪 → 正常
- `drawImage(ImageBitmap)` 正常，`drawImage(HTMLImageElement)` 对 clipboard blob URL 彻底无效

**项目规则（记入 MEMORY.md）：**
- 不再用 `drawImage(HTMLImageElement)` 处理 blob URL 图片
- 图片处理统一走 `fetch + createImageBitmap` 路径
