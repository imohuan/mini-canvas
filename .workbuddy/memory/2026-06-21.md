## BaseNode 标题栏宽度反缩放

`BaseNode.vue` 标题容器用 `left-1 right-1` 撑满节点，但 `createCappedStyle` 套了 `scale(1/zoom)` + `transform-origin: left top`。zoom>1 时缩放从左上角收缩，标题视觉宽度只剩节点 1/zoom，右侧空一大截。

### 改动
- `src/canvas/core/utils/viewportSpace.ts`：`CappedStyleOptions` 新增 `width?: number`。`createCappedStyle` 在启用 scale 时把布局宽度按 `1/inv`(=max(zoom,1)) 放大，scale 后视觉宽度回到原始值。与 `cardFrameStyle` 的"放大布局再 scale 回来"同一套路。
- `src/canvas/core/components/Decoration/BaseNode.vue`：`titleTransformStyle` 传 `width: cardWidth.value - 8`；模板标题 div 去掉 `right-1`（设了 width 后 right 被忽略，删掉避免歧义），保留 `left-1`。

### 关键点
- zoom≤1 时 inv=1，width 不放大，行为与原 `left-1 right-1` 完全一致（封顶只在 zoom>1 触发）。
- 左右内边距对称：left=4px(node DOM)=4*zoom(屏幕)，width 补偿后 right 也是 4*zoom(屏幕)。
- layout 框放大可能超出节点，但 scale 后视觉收回；与已有 cardFrameStyle 同模式，vue-flow 不受影响。
- `vue-tsc --noEmit` 通过。

## PanoramaNode 上传逻辑改造：基于端口连接

将全景节点的图片显示从"自身 data.imageUrl"改为"上游连接的 image 节点 imageUrl"。

### 改动文件
- `src/canvas/core/nodes/panorama/PanoramaNode.vue`
- `src/canvas/core/nodes/panorama/PanoramaNodePlugin.ts`

### 核心设计
1. **显示**：PanoramaNode 用 `useVueFlow()` 拿响应式 `getEdges`/`findNode`，computed `connectedImageUrl` 查找 `target===nodeId && targetHandle==='target'` 的边，取 source 节点 `data.imageUrl`。watch 该 computed 驱动 three.js 纹理更新。
2. **上传**（`handlePanoramaUpload`）：不再写 panorama 自身 data，而是创建新 image 节点（位置在 panorama 左侧，`panoPos.x - panoCardWidth - 80`），保存资产，读尺寸 fitCardSize，`addNodes` + `nextTick` + `addEdges`（source=image, target=panorama, sourceHandle='source', targetHandle='target'）。
3. **重置**（`handlePanoramaReset`）：删除所有 `target===nodeId && targetHandle==='target'` 的边，预览自然清空（computed 变空 → destroyThree → 重建空场景）。
4. **下载**（`handlePanoramaDownload`）：从上游 image 节点取 imageUrl/imageName。
5. **双击进编辑**：判断改为 `context.actions.getEdges().some(e => e.target===nodeId && e.targetHandle==='target')`。

### 关键点
- 新建 image 节点必须设 `sourcePosition: Position.Right` + `targetPosition: Position.Left`，否则 BaseNode 不渲染 handle 无法连出/连入。
- `useVueFlow()` 无 id 调用能拿到当前实例（与 BaseNode 一致）。
- vue-tsc 类型检查通过。

### 遗留
- 多次上传会创建多个 image 节点（未做替换旧连接逻辑），如需可后续迭代。
- CanvasNodeData.ts 的 PanoramaNodeData.imageUrl/panoUrl 字段保留未删（向后兼容旧数据）。

## 连线拖拽菜单逻辑迁移设计

针对"从端口拖到空白弹菜单"功能，做架构设计+实现计划。

### 关键发现
1. `useCanvasConnection.ts` 是零引用 dead code，Canvas.vue 自己重写了一份完整版
2. `connectionState` 全部在 Canvas.vue 写入（onConnectStart / mousemove / onConnectEnd 等 30+ 处）
3. `ContextMenuPlugin` 已监听 `connectionContextMenu` 事件负责菜单 UI，但临时节点创建在 Canvas.vue，清理在插件——割裂
4. 判定 bug：松开在节点上仍弹菜单、吸附后仍弹菜单

### 设计决策
- 状态归属：`ConnectionState` 放 `useCanvasStore`（不分散到插件 reactive）
- 行为归属：ContextMenuPlugin 监听 `connectEnd` 做判定+创建临时节点，闭环临时节点生命周期
- 入口归属：Canvas.vue 只写"原始事实"（鼠标位置、源信息），删除业务判定
- `PluginContext` 扩展 `connectionState` / `isConnecting` / `canShowConnectionMenu` 直接引用

### 字段重构
- 合并 `isConnecting + sourceNodeId + sourceHandle` → `activeConnection`
- 合并 `hoverFeedback* + invalidFeedback*`（6 字段）→ `hoverNode`（status: 'valid'|'invalid'）
- 新增 `hoverTarget` / `snapTarget` / `tempConnection` / `mouseFlowPosition` / `mouseScreenPosition`
- 字段数 9 → 8，语义更清晰

### 修复机制
`canShowConnectionMenu = isConnecting && hoverTarget.type === 'pane' && !snapTarget.isSnapped && !tempConnection`
